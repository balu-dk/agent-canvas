Proposal — Auto-Discover Agent Conversations

SDK aggregates sessions from all ACP agents into one conversation list — canvas just renders it

Builds on top of PR #421 (mini setup server + multi-select backend)  ·  #405

1 Architecture

The SDK (agent-server) is already the single owner of the conversation list. Instead of adding new frontend-facing endpoints, we tell the SDK which ACP agents to connect to at startup. The SDK calls list_sessions() on each, merges everything into its existing conversation list, and the frontend just renders what it already gets.

Key insight: The SDK already owns the conversation list. We don't need new /api/agents or /api/sessions endpoints. The SDK connects to ACP agents at startup, calls list_sessions() on each, and merges discovered sessions into its existing list. The frontend changes are minimal — just render the agent_id that's now on each conversation.

What powers this — each backend responds to GET /api/sessions:

// GET http://localhost:18000/api/sessions?cwd=/home/user/projects [ { "session_id": "conv-001", // already tracked → shows as normal card "agent_id": "openhands", "status": "running", "title": "Fix login bug", "created_at": "2026-05-12T23:50:00Z" }, { "session_id": "cli-abc-123", // NOT tracked → "Discovered from CLI" "agent_id": "openhands", "status": "paused", "title": "Migrate database schema", "created_at": "2026-05-12T22:30:00Z" } ] // GET http://localhost:18002/api/sessions?cwd=/home/user/projects [ { "session_id": "conv-002", // already tracked "agent_id": "claude-code", "status": "running", "title": "Write API docs", "created_at": "2026-05-12T23:45:00Z" }, { "session_id": "cli-xyz-789", // NOT tracked → "Discovered from CLI" "agent_id": "claude-code", "status": "paused", "title": "Debug payment flow", "created_at": "2026-05-12T23:28:00Z" } ]
The key idea: Canvas calls GET /api/sessions on every connected backend, diffs the results against its own conversation list, and surfaces untracked sessions below a "Discovered from CLI" divider. Clicking a discovered session creates a canvas conversation that attaches to the existing ACP session via POST /api/sessions/{id}/resume.

4 SDK-Side Changes

The SDK (agent-server) needs two additions: accept ACP agent config at startup, and merge their sessions into its conversation list.

4a. Startup config — tell the SDK which ACP agents to connect to

# config.toml (or however the SDK is configured) [[acp_agents]] id = "openhands" endpoint = "http://localhost:3000" # ACP protocol endpoint [[acp_agents]] id = "claude-code" endpoint = "http://localhost:3001" [[acp_agents]] id = "codex" endpoint = "http://localhost:3002"

4b. SDK calls list_sessions() on each ACP agent

No new endpoints. The existing GET /api/conversations just gains two optional fields: agent_id (which ACP agent owns this session) and source ("canvas" or "cli"). The frontend uses these to render agent tags and "Discovered from CLI" sections.

5 Frontend Changes

# What Details
1 Frontend Agent tag on ConversationCard Read agent_id from the conversation response. Render a small colored badge (e.g. OpenHands Claude Code Codex) next to the conversation title in the header row.
2 Frontend "Discovered from CLI" section Conversations where source === "cli" are grouped below a dashed divider in the conversation panel. Clicking one triggers resume_session() on the SDK, which attaches to the existing ACP session.
3 Frontend Remove ChooseAgentStep Delete the static agent selection step from onboarding. Onboarding becomes 3 steps: Choose Backend → Setup LLM → Say Hello.
4 Frontend Update AppConversation type Add optional agent_id?: string and source?: "canvas" | "cli" to the conversation type. Map agent_id to display name + color via a simple lookup table.

6 Implementation Phases

Phase A — SDK: ACP agent aggregation SDK

  1. Add acp_agents config to SDK startup
  2. On startup (and periodically), call list_sessions(cwd) on each connected ACP agent
  3. Merge discovered sessions into the existing conversation list, tagging with agent_id and source
  4. When the user clicks into a discovered session, call resume_session() on the ACP agent

Phase B — Frontend: agent tags + CLI section Frontend

  1. Add agent_id / source to AppConversation type
  2. Render agent badge on ConversationCardHeader
  3. "Discovered from CLI" divider + dashed-border cards for source === "cli"
  4. Remove ChooseAgentStep from onboarding, move Choose Backend to Step 0

7 Open Questions

  1. ACP agent config delivery: Is config.toml the right place, or should the canvas send ACP agent endpoints to the SDK via an API call at setup time?
  2. Polling interval: How often should the SDK call list_sessions() on each ACP agent? On every GET /api/conversations request, or on a background timer?
  3. Session resumption: When the user clicks a CLI-discovered session, should the canvas replay the full event history or show a "resumed" marker and start fresh?
  4. Agent display names: Should the SDK return a display name + icon key alongside agent_id, or should the frontend maintain a static mapping?
  5. Cross-agent "new conversation": When the user starts a new conversation, should they pick which agent to use? Or default to a configured primary agent?

Summary

  • SDK adds: acp_agents startup config; internal list_sessions() polling; session merging into conversation list
  • SDK extends: GET /api/conversations response with agent_id and source fields
  • Frontend adds: agent tags on conversation cards; "Discovered from CLI" section; removes Choose Agent step
  • No new endpoints: SDK is the aggregation layer; frontend just reads the existing conversation API
  • 2 phases: (A) SDK aggregation → (B) Frontend rendering
